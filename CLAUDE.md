# WatchNext — Claude.md

## Project

Shared movie & TV recommender for two-person households backed by Trakt integration and AI recommendations.

## Tech Stack

- **Frontend**: Flutter 3.11+ (Dart) with Material 3 dark theme
- **State management**: Riverpod 2.5 (providers + StateNotifier)
- **Routing**: go_router 14.2 with typed routes
- **Backend**: Firebase (Auth, Firestore, Cloud Functions, Cloud Messaging)
- **Cloud Functions**: Node 22 TypeScript, firebase-admin 13, firebase-functions 7, Anthropic SDK 0.30
- **External APIs**: 
  - Trakt (watch tracking, OAuth 2.0, history/ratings/recommendations)
  - TMDB (movie/TV metadata)
  - Claude API (Phase 7+, taste profiles)
  - Reddit JSON (Phase 7+, social proof)
- **Storage**: SharedPreferences (device-level mode toggle), Firestore (all household data)

## Key Conventions

### State Management
- **Providers**: `Provider<T>` for singletons, `StreamProvider<T>` for Firestore listeners, `FutureProvider<T>` for one-shot async
- **StateNotifier**: Used only for mutable user state (`ModeController`)
- **Naming**: `*Provider` for providers, `*Service` for singletons, `*Controller` for StateNotifier
- **Autodipose**: StreamProviders handling screen-scoped data use `.autoDispose` to clean up listeners on pop

### Firestore Collections & Rules
```
households/{householdId}/
  ├─ members/{uid}              # User ref + Trakt tokens + predict counters
                                # Counters: predict_total / predict_wins (legacy, pre-split context);
                                #           predict_total_solo / predict_wins_solo;
                                #           predict_total_together / predict_wins_together
                                # trakt_history_scope: 'shared'|'personal'|'mixed' (default mixed)
                                #   stamps Rating.context on imported Trakt ratings
  ├─ watchEntries/{mediaType:tmdbId}
  │  └─ episodes/{traktId}      # Per-ep timestamps
  ├─ ratings/{id}               # {uid, level, target_id, stars, tags, note, rated_at, pushed_to_trakt, context?}
                                # context: 'solo'|'together'|null (null = legacy / Trakt historical)
  ├─ watchlist/{scope:owner_or_shared:mediaType:tmdbId}
                                # {mediaType, tmdbId, scope, owner_uid?, addedAt, ...}
                                # scope: 'shared' (both members) | 'solo' (owner_uid only)
                                # shared → owner_or_shared = 'shared'; solo → owner_or_shared = uid
  ├─ predictions/{mediaType:tmdbId}
                                # {entries: {uid: {stars, skipped, submitted_at, context?}}, reveal_seen, ...}
                                # entry.context: 'solo'|'together'|null (determines counter routing)
  ├─ recommendations/{id}       # {mediaType, tmdbId, match_score, match_score_solo, ai_blurb, ...}
  ├─ tasteProfile/{id}          # {combined, per_user, per_user_solo, per_user_together, member_uids, updatedAt}
                                # per_user_solo[uid]: built from ratings with context='solo' OR null
                                # per_user_together[uid]: built from ratings with context='together' OR null
                                # per_user[uid]: cross-context (kept for back-compat + legacy consumers)
  ├─ decisionHistory/{id}       # "Decide" screen picks
  ├─ conciergeHistory/{id}      # Concierge chat logs
  ├─ collections/{id}           # Custom lists (Phase 10+)
  └─ issueBatches/{batchId}     # "Report an issue" queue docs. Per-submitter. `status`: pending|dispatched|cancelled. Client-writable only to set `status = cancelled`; the rest is admin-SDK-only.
users/{uid}/
  └─ {householdId}              # Pointer for O(1) household lookup post-auth
invites/{token}
  └─ Lookup table: {householdId, createdBy, expiresAt}
redditMentions/{id}             # Global; written by CF
rescoreQueue/{householdId}      # Background re-score marker. Admin-SDK only.
                                # {household_id, dirty_since, last_scored_at}
                                # dirty_since > last_scored_at → drain picks it up
households/{householdId}/badges/{badgeId}
                                # Server-persisted badge state. Admin-SDK writes only.
                                # {id, name, progress, target, earned, member_uid?, earned_at?, updated_at}
                                # badgeId = 'century_club' | 'genre_explorer' | 'prediction_machine_{uid}'
```
- All reads/writes scoped to household membership via `isMember(householdId)` check
- Two-person cap enforced client-side (`HouseholdService.joinByInviteCode`)
- Trakt OAuth tokens encrypted at `members/{uid}.trakt_refresh_token`

### Screen/Widget Structure
- **Screens**: Consume providers via `ref.watch()`, emit side effects via `ref.read().method()`
- **Shared widgets**: Stateless except `ScaffoldWithNavBar` (manages share intent listeners at root)
- **Error handling**: `AsyncErrorView` + per-provider error states + global error widget fallback
- **Bottom sheet**: `showModalBottomSheet()` for confirm dialogs; persistent on navigation
- **Share-to-Save** (Phase 4): `warm_start` + `cold_start` listeners in `ScaffoldWithNavBar` parse Android SEND intents

### External API Integration
- **Trakt**: Client ID as `--dart-define TRAKT_CLIENT_ID`, secret only in Firebase Secrets Manager
  - OAuth browser flow via `flutter_web_auth_2`, CSRF state saved to Firestore
  - Token refresh auto-triggered on 401; revoke on household leave
  - Callable proxies: `traktExchangeCode`, `traktRefreshToken`, `traktRevoke` in Cloud Functions
- **TMDB**: Key via `--dart-define TMDB_API_KEY`, fetches metadata on demand
- **Claude**: Async callable `concierge` + `generateTasteProfile`, AI blurbs cached in `/recommendations`

### Secret Handling
- Keys passed via `--dart-define-from-file=env.json` (build-time) or Firebase Secrets Manager (runtime)
- `env.json` gitignored; use `env.example.json` template
- Trakt `client_secret` lives **only** in `TRAKT_CLIENT_SECRET` Firebase secret
- No hardcoded secrets; all validated at startup

## Known Gotchas

1. **Cloud Functions region**: All CF callables explicitly set to `europe-west2` (London) to co-locate with Firestore; client code uses `FirebaseFunctions.instanceFor(region: 'europe-west2')` — do not default to us-central1
2. **Firestore model parsing**: Hardened to skip missing fields and map null → sensible defaults; see `fromDoc` factory methods. Partial documents from share-parser are normal
3. **Trakt token refresh**: Happens auto on 401; client must catch and retry. Token stored encrypted at member doc — check `firebase_auth_mocks` in tests for token mocking
4. **Two-person cap**: Enforced in service layer (line-of-code comment in rules acknowledges Firestore can't count siblings easily)
5. **Share-to-Save listeners**: Cold-start listener set up in `main.dart`, warm-start in `ScaffoldWithNavBar`; both parse Android SEND intents asynchronously
6. **Phase 7+ AI blurbs**: `Recommendation` doc has dual fields (`ai_blurb` for together, `ai_blurb_solo` for solo mode); only populated once taste engine runs
7. **TMDB genres**: Hardcoded map in `utils/tmdb_genres.dart` — update if TMDB API schema changes
8. **Debug keystore**: Pinned in `android/app/build.gradle`; SHA-1 verified in CI — restores on each checkout to keep Google Sign-In consistent
9. **ExpansionTile + flag-gated nested Consumer**: avoid `children: [if (_flag) Consumer(...)]`. The Consumer never observes the FutureProvider's resolution because the rebuild after `setState` races the tile's expansion animation; symptom is a spinner that never resolves. Instead split into a sticky-flag wrapper + separate `ConsumerWidget` body — see `_GenreRow`/`_GenreRowBody` in `discover_screen.dart`.
10. **Home filter stack**: eight knobs compose on the stream — genre set (free-form TMDB multi-select via `selectedGenresProvider`), runtime bucket, `YearRange {minYear?, maxYear?}`, media type (`MediaTypeFilter` via `mediaTypeFilterProvider`, per-mode persisted under `wn_media_type_solo/_together`), Oscar-winners-only (`oscarFilterProvider`, per-mode persisted under `wn_oscar_winners_solo/_together`), Exclude-animation (`excludeAnimationProvider`, per-mode persisted under `wn_exclude_animation_solo/_together`), Sort mode (`sortModeProvider` — one of `topRated` | `popularity` | `recent` | `underseen`, per-mode persisted under `wn_sort_mode_solo/_together`), and Curated source (`curatedSourceProvider` — currently `none` | `criterion` via TMDB company id 1771, per-mode persisted under `wn_curated_source_solo/_together`). Mood pills have been retired — the genre set is the canonical filter and mood presets were judged redundant with it. When the user narrows any way (genre/year/runtime/media type/oscar/exclude-animation/sort/curated), `recommendations_service.refresh()` fires `tmdbService.discoverPaged()` for the relevant media types; when media-type is set it only fires the matching one; when oscar is on it passes `keywordIds: [210024]` (TMDB's "oscar-winning-film" keyword) to `discoverPaged` and stamps `is_oscar_winner=true` on the resulting rows; when exclude-animation is on it passes `excludeGenreIds: [16]` → `without_genres=16` server-side AND drops animation-tagged rows from trending/top_rated inside `buildCandidates` (belt-and-braces because TMDB's `with_genres` is OR-of-inclusion — a crime-and-animation title matches `with_genres=80`); when sort mode is `underseen` it passes `maxVoteCount=500` → `vote_count.lte=500` and suppresses the trending/top_rated baseline inside `buildCandidates`; `recent` picks a media-type-specific sort token (`primary_release_date.desc` for movie, `first_air_date.desc` for tv) via `SortMode.tmdbSortBy(mediaType)`; `popularity` uses `popularity.desc`; when curated source is `criterion` it passes `withCompanies='1771'` AND suppresses baseline (trending/top_rated would dilute a curator pool). If the user explicitly picked Animation as a genre, the exclude toggle is a no-op (self-contradictory). Discover rows land in `buildCandidates` under `source='discover'` with `discoverCap=40` vs trending/top-rated's `tmdbCap=10`. The filter widgets live inside a single `_FiltersPanel` ExpansionTile on the Home screen. Panel body is flat (no horizontally-scrolling FilterChip rows): small uppercase section labels (`_FilterSectionLabel`) sit above a Genre InputChip row, four `SegmentedButton`s (`_MediaTypeSegment` / `_RuntimeSegment` / `_SortModeSegment` / `_CuratedSourceSegment`) sharing a common `_segmentStyle()`, the `YearRangeSlider`, and three label-left + trailing-Switch rows (`_FilterSwitchRow`) for Oscar winners / Exclude animation / Include watched. Media and Runtime both carry an explicit `null`-valued "Any" segment so deselect goes through the same `SegmentedButton.onSelectionChanged` path. Don't reintroduce per-option FilterChips — the flat look was an intentional UX shift away from pill-heavy controls. Runtime filter is **strict**: `bucket.matches(r.runtime)` — unknown-runtime recs drop out. To keep the pool non-empty, `discoverPaged` uses TMDB's `with_runtime.gte/lte` server-side filter when a bucket is active, and the service stamps a representative runtime on each discover row (TMDB confirmed it's in-bounds). Trending/top_rated lack runtime and are intentionally excluded when a runtime bucket is active — we can't honestly show them under a specific length. Media type filter does strict equality `r.mediaType == filter.recMediaType`; cleared filter shows both. Year filter drops unknown-year items when any bound is set. Oscar filter is also strict: only recs the discover pass tagged with `is_oscar_winner=true` survive — trending/top_rated rows that haven't been confirmed via the keyword discover drop out, even if they happen to be real Oscar winners. Exclude-animation filter is client-side strict too: `!r.genres.contains('Animation')`. Sort mode + curated source don't add a client-side filter — they reshape the server-side pool; their contribution is carried entirely by the TMDB query params + baseline-suppression flags.
11. **SharedPreferences test caching**: `SharedPreferences.getInstance()` caches a singleton across tests in the same isolate, so calling `setMockInitialValues(...)` after an earlier test has already called `getInstance()` won't take effect on subsequent reads. Fix: after `setMockInitialValues`, either call `prefs.reload()` to refresh the cached instance, or construct a controller with an explicit prefs + override the StateNotifierProvider directly. See `test/widgets/genre_sheet_test.dart` for the override pattern (includes overriding `viewModeProvider` — otherwise `selectedGenresProvider` reads from the wrong mode).
12. **TMDB request timeout**: `TmdbService` wraps every `_get` in `.timeout(kRequestTimeout)` (15s default). `http.Client` has no built-in timeout and `discoverPaged` can fan out to ~18 TMDB calls on a single pull-to-refresh; any stuck request previously hung the refresh spinner indefinitely. Timeouts surface as `TimeoutException`, which the service layer's `_safeTmdb` wrapper swallows into an empty payload so a single flaky request doesn't blank the whole candidate pool. The constructor accepts a `timeout` override for tests.
13. **Two-phase refresh**: `RecommendationsService.refresh()` returns as soon as the Firestore batch write completes (Phase A — typically <5s); the `scoreRecommendations` callable + optional taste-profile refresh run in `unawaited(_backgroundScore(...))` (Phase B — 60-120s). Pull-to-refresh UX: spinner stops on Phase A so Home lights up immediately with default-scored recs; Claude's real scores arrive via Firestore stream updates later. New rec keys are seeded with `match_score=50, scored=false, ai_blurb=''`; existing keys skip those fields on merge so previously Claude-scored recs don't visibly regress. If Phase B fails (network, CF cold start, auth), the scheduled `processRescoreQueue` CF (every 10 min) picks up the slack — that's why `_backgroundScore` swallows errors into `developer.log`. **Widened pool**: `discoverPaged` defaults to `poolFloor=40, maxPages=5`; the CF `MAX_CANDIDATES=100` — upped from 20/3/50 so narrow filters (e.g. "War, 1970-1989") still yield a sizable list after client-side filters run. `drainPages` retries the next page on per-page errors (doesn't abort the whole rung).
14. **Stream window sized for Phase A**: `/recommendations` is streamed with `.limit(300)` — deliberately wider than the top-120 a user sees. Phase A writes fresh discover rows at `match_score=50`; if legacy Claude-scored (60–90) recs filled the first 120 slots, a narrow filter would see an empty list until Phase B re-ran. Wider window keeps just-written discover rows visible to the client-side filter from the instant Phase A lands. `writeCandidateDocs` reads up to 800 existing ids to preserve scores on merge.
15. **Home auto-refresh on filter change**: the Home screen's `build()` registers `ref.listen`s on `selectedGenresProvider`, `runtimeFilterProvider`, `yearRangeProvider`, `mediaTypeFilterProvider`, and `oscarFilterProvider`. Any change schedules a 700ms-debounced non-forcing refresh (`refreshRecommendationsProvider(false)`) so the pool is rebuilt against the current filter state without the user having to pull-to-refresh. The debounce Timer is cancelled in `dispose()`. Compare sets with `setEquals` from `package:flutter/foundation.dart` — the raw `!=` comparison on `Set` is reference equality and would fire spuriously.
16. **Concierge SDK path**: `@anthropic-ai/sdk@0.30.1` doesn't accept `cache_control` on the stable `messages.create` `TextBlockParam` — only the deprecated `beta.promptCaching.messages.create` path does. Concierge is low-frequency (one chat at a time) so we intentionally use the stable endpoint without `cache_control` rather than ride the beta namespace. The Claude call is wrapped in try/catch so upstream errors surface as an `HttpsError("internal", <reason>)` with the original error message; the client parses `FirebaseFunctionsException` and shows the reason in the chat bubble instead of a generic "something went wrong".
18. **Callable result maps on Android**: `HttpsCallableResult.data` comes back as `Map<Object?, Object?>` from the Android platform channel (iOS varies too). `result.data as Map<String, dynamic>` compiles but throws `type 'Map<Object?, Object?>' is not a subtype of type 'Map<String, dynamic>'` at runtime; `.cast<Map<String, dynamic>>()` on a list of nested maps throws the same at iteration time. Always normalize with `Map<String, dynamic>.from(result.data as Map)` at the top level AND `Map<String, dynamic>.from(m)` per nested map before constructing models. `ConciergeService.chat` hit this only on Like-these responses (chat-only paths often have empty titles arrays, hiding the bug).
19. **"More like these"**: `/home/ubuntu/projects/watchnext/lib/screens/like_these/like_these_sheet.dart` reuses the `concierge` CF with a synthetic prompt — no separate endpoint. The sheet collects 2–8 TMDB seed titles via `searchMulti`, builds a "suggest titles that fit this group as a whole" message, and sends it with a fresh `sessionId` (so the concierge doesn't treat it as a continuation of an earlier chat) and empty `history`. Response titles render as tappable tiles that navigate to `/title/:mediaType/:tmdbId`. Entry point: "Like these" OutlinedButton in the Home action row next to "Surprise me". Also see TitleDetail's **Similar titles** carousel, which surfaces TMDB's `/similar` results (free — already appended via `append_to_response=similar`).

20. **Claude hallucinates tmdb_ids**: the concierge prompt asks for a `tmdb_id` per suggestion, but Claude will confidently emit an id that belongs to a different movie — we've seen "The Shining" come back with Ice Age's id, so the poster rendered Ice Age and taps opened Ice Age. `ConciergeService.chat` runs every suggestion through `TmdbService.searchMulti(title)` client-side, prefers year+media_type-matching results, and replaces `tmdbId` + captures `posterPath` with real TMDB data. Suggestions that don't resolve are dropped rather than shown as broken rows. No server-side fix because the Cloud Function doesn't have a TMDB key configured (redditScraper's `process.env.TMDB_API_KEY` lookup is a silent no-op without it).

21. **Stremio / IMDb deep links**: TitleDetail shows "Stremio" + "IMDb" buttons when an imdb_id resolves. Movies carry `imdb_id` at the top level of TMDB's `/movie/:id` payload; TV does **not** — we append `external_ids` to `tvDetails` so `external_ids.imdb_id` is available. Schemes: `stremio:///detail/{movie|series}/{imdb_id}/{imdb_id}` (path repeats per Stremio's routing) and `imdb:///title/{imdb_id}/`; both fall back to `https://web.stremio.com/#/detail/...` / `https://www.imdb.com/title/...` when the app isn't installed. AndroidManifest declares both schemes under `<queries>` — without it `canLaunchUrl` always returns false on Android 11+. `url_launcher` is the only dep added.

22. **"Watching" definition duality**: `watchedKeysProvider` (used by `includeWatchedProvider` to hide titles from suggestion surfaces) treats ANY watchEntry as "seen" — finished AND in-progress. The rationale: don't recommend a show the household is partway through. Separately, the Watchlist screen's `Watching` segment pulls directly from `watchEntriesProvider.where((e) => e.inProgressStatus == 'watching')` so in-progress TV surfaces there even when it was never saved to the watchlist. Don't confuse the two: the provider is a *skip set*, the watchlist segment is a *display list*.

23. **Stremio addon**: `functions/src/stremio.ts` exports an `onRequest` HTTP function that implements the Stremio addon protocol and two onCall siblings (`provisionStremioToken`, `revokeStremioToken`). Token→household mapping lives in `/stremioTokens/{token}` with `{household_id, uid}` (admin SDK only — Firestore rules deny all client access). URL shape: `https://europe-west2-watchnext-9920f.cloudfunctions.net/stremio/{token}/manifest.json`. Requires the **TMDB_API_KEY** Firebase secret because WatchlistItems don't carry imdb_id and Stremio keys everything by imdb; the addon resolves missing ids via TMDB `/external_ids` and caches them back onto the watchlist doc. **Only shared-scope watchlist items surface** — solo items stay private to their owner even though tokens are per-user. Catalog id is `wn_watchlist` with both `movie` and `series` types declared in the manifest. Today only the watchlist catalog ships; recommendations + next-up catalogs are on the Stremio track in ROADMAP.md. The install URL uses `https://`; client-side we swap to `stremio://` to trigger install. Invocation requires `invoker: "public"` — Stremio is unauthenticated; the token is the only auth mechanism. CORS is opened via `cors: true` since Stremio web loads the addon in a browser sandbox.

24. **TMDB merge order — discover leads, baseline follows**: `buildCandidates` puts `discoverMoviesPayload` and `discoverTvPayload` AHEAD of trending/top_rated in the dedup walk. Reason: when a row appears in both discover and trending, the discover tag carries the user's narrowing intent (genre, year, runtime, oscar keyword) — and for the Oscar filter specifically, the discover row carries `is_oscar_winner=true`. If trending won dedup, a current-year Best Picture that's also trending this week would lose the oscar tag and disappear under the Oscar filter. Tests in `recommendations_service_test.dart` (`buildCandidates — discover payloads`, `buildCandidates — oscar discover tagging`) lock this contract.

25. **Sticky `is_oscar_winner` + `curator` on merge**: `writeCandidateDocs` only writes the `is_oscar_winner` field when the candidate map carries `true`. False/missing values are intentionally omitted so Firestore's `merge: true` semantics leave any previously-written `true` untouched — a movie doesn't un-win an Oscar between refreshes. Mirror logic in `buildCandidates`: it only stamps `is_oscar_winner: true` on discover rows when `discoverIsOscar=true`, never `false`. Same pattern for `curator` (e.g. `'criterion'`): set only when `discoverCurator` is non-empty on a discover row; empty-string and missing are both ignored by `writeCandidateDocs` so a prior `curator: 'criterion'` survives merge with a non-curator refresh. Both tags drive strict client-side filters on Home — Oscar filter keeps only `isOscarWinner=true`, Curated filter keeps only `curator == CuratedSource.name` — which prevents stale non-matching recs from leaking through when a filter is active. So the round trip from "user toggled curator on → refresh → tag written" is preserved across subsequent refreshes that don't include the keyword/company filter.

26. **Decide "Surprise me" exploratory rung**: `DecideController.rerollExploratory()` is opt-in (button-driven from the Negotiate UI), separate from the standard `rerollCandidates` shuffle. It samples a random pre-2020 decade from `kExploratoryDecades` (`(1970,1979), (1980,1989), (1990,1999), (2000,2009), (2010,2019)`), fires `tmdbService.discoverPaged` for both movie + tv with the decade as date bounds + `vote_count.gte=300` for movies / `100` for tv (older catalog has thinner TV at high vote counts), interleaves the results, excludes already-watched + currently-shown titles, and replaces the negotiate pool with the first 5. Excludes the current decade so it doesn't just resurface what trending already covers. Failure modes (empty TMDB response, thrown exception) leave the existing pool intact and surface an error containing the decade so the UI can explain. Does NOT touch the Decided-screen `reroll` flow or the Home recommendation pool — purely a Decide-only exploratory tool. Tests in `test/providers/decide_reroll_exploratory_test.dart`. The `decadePicker` constructor param on `DecideController` is for test determinism; production uses the random picker.

27. **Firestore dot-notation is `.update()`-only**: `ref.set({'a.b': v}, SetOptions(merge: true))` stores a **literal field named "a.b"** with a dot in it — it does NOT write the nested path `a.b`. Only `ref.update({'a.b': v})` interprets the dot as a nested path. This bit `WatchEntryService` twice (mark watched didn't persist; unmark watched didn't clear) before it was caught. If you need to mutate a nested map field, use `.update()`. If the doc may not exist, `.get()` first and early-return — `update()` throws on missing docs, whereas `set(merge:true)` silently creates them but won't honour dot-notation. Tests in `test/services/watch_entry_service_test.dart` explicitly assert `doc.containsKey('watched_by.<uid>') == false` so a regression to `.set(merge:true)` fails the suite loudly.

28. **3-state watch status on TV title detail**: `TitleDetail` renders a `SegmentedButton<_WatchStatus>` (Not / Watching / Watched) for TV titles and keeps the classic 2-state Mark-watched FilledButton.tonal for movies ("watching" isn't meaningful for a 90-120m sitting). The current state is derived from `watchedBy[uid]` first (watched wins), then `inProgressStatus == 'watching'`. Selection changes route to `_setWatchStatus` which picks the right service method: `markWatched` / `unmarkWatched` / `markWatching` / `unmarkWatching`. Choosing "Watched" while "watching" additionally clears the in-progress flag so the status doesn't stick around as a ghost chip.

29. **Inline trailer on title detail**: `TmdbService.movieDetails` / `tvDetails` now append `videos` to `append_to_response` so the detail payload carries `videos.results`. `_TrailerSection` on TitleDetail renders an idle "Watch trailer" OutlinedButton that expands into a full-width inline `YoutubePlayer` on tap (via `youtube_player_flutter`). Trailer selection preference order (`pickTrailerKey` — unit tested in `test/screens/pick_trailer_key_test.dart`): official YouTube Trailer → any YouTube Trailer → any YouTube Teaser → first YouTube video. Section stays silent when no YouTube video is attached (older catalog) rather than showing a broken button. The YoutubePlayerController is disposed in `_TrailerSectionState.dispose()` to avoid leaking the WebView when the detail screen pops. `test/services/tmdb_service_test.dart` locks the `append_to_response` strings to `credits,keywords,similar,videos` (movies) and `credits,keywords,similar,external_ids,videos` (tv) so the videos append can't silently be dropped.

30. **"Upcoming for you" Home section**: `upcomingForYouProvider` (in `lib/providers/upcoming_provider.dart`) drives a 230-tall horizontal poster row between Tonight's Pick and the main rec list. Fetches `/movie/upcoming` + `/tv/on_the_air` in parallel, filters out titles whose key appears in `watchedKeysProvider` (which includes in-progress — so the household isn't shown stuff they're already watching), then ranks by genre-overlap against the per-mode taste profile (`per_user_solo` / `per_user_together`, falling back to `per_user` then `combined`). Deliberately client-side only — not round-tripped through the Phase 7 scoring CF because TMDB's upcoming feed rotates too fast and the carousel's job is surfacing what's *coming*, not predicting 5-star hits. Ranking uses `_scoreGenreOverlap` (sum of taste weights per candidate genre, clamped to [0,1], × 100 → int); ties break on soonest release date. Returns empty → silent render (no loading/error UI) so a flaky TMDB call never blanks the Home screen. Limit 20. TV has no first-run-only feed on TMDB, so `/tv/on_the_air` (active within a week) is the analogue — expect some current-run TV to bleed through alongside strictly-upcoming titles.

31. **Refresh-race epoch guard**: `RecommendationsService` owns an `int _refreshEpoch = 0` counter. At the top of `refresh()` we capture `final myEpoch = ++_refreshEpoch;`. Before the Firestore write AND before kicking off background Claude scoring we check `if (myEpoch != _refreshEpoch) return;` — a newer refresh has superseded this one, so bail silently. Without this guard, an older refresh whose TMDB fan-out (18 calls on a narrow filter) returned slowly could complete its `writeCandidateDocs` after a newer refresh's already-committed pool, overwriting the fresher results — the "selected filters show a set of options that get overwritten a few seconds later" symptom. Both refreshes still run their fetches to completion; we only short-circuit the mutation side so the *last-started* refresh wins regardless of which finishes TMDB first. Tests in `test/services/recommendations_service_test.dart` under `RecommendationsService.refresh — epoch guard` inject a slow-returning TMDB mock for refresh 1 and a fast one for refresh 2, then assert refresh 1's candidates never land in Firestore.

32. **Flattened title-detail action row**: The old Wrap of mixed FilledButton / FilledButton.tonal / OutlinedButton variants has been replaced with a uniform grid of compact `OutlinedButton.icon` pills (36px min height, 10px horizontal padding, 12px font, `VisualDensity.compact`) — Watchlist, (3-state SegmentedButton for TV) Watched toggle for movie, Rate — followed by a `_PredictOverflowMenu` (⋯) that hosts Predict + See Reveal behind a popup, then a trailing strip of `_ExternalLinkButton` tiles (icon + tiny 10.5pt label) for the external-link secondaries ("Stremio" and "IMDb"). Labels on the external strip were added because icon-only made the deep-links invisible; they stay intentionally small so the strip remains visually subordinate to the primary pills. **Predict was demoted from a primary pill**: its pre-watch-commit + post-watch-return flow has too much friction for the primary action row. The overflow menu renders only when `canPredict || canReveal`, and shows a small accent dot on the ⋯ icon when `canReveal` is true (a Reveal is actually waiting — the only state that warrants pulling the user in). Don't promote Predict back without rethinking the flow; `/predictions` data, the Reveal screen, and the Prediction Machine badge all still work through the overflow entry. Single `_ActionRow` widget still owns the whole row; don't reintroduce the mixed-variant styling when adding new actions.

33. **External ratings section on title detail**: `_RatingsSourcesSection` sits right under the trailer and shows TMDB `vote_average` (formatted `X.X / 10`) + `vote_count` inline, followed by deep-link chips: IMDb reviews (`/title/{imdb}/reviews/`, only when `imdb_id` resolves), Rotten Tomatoes search, Letterboxd search (movie-only). No OMDb dependency — keeps the ratings surface free of another API key / rate-limit. Chips use `_RatingLinkChip` (compact OutlinedButton + `open_in_new` icon) which escapes the title via `Uri.encodeComponent`. If both `vote_average` and `title` are missing the section silently collapses to `SizedBox.shrink()`.

34. **Trailer-playing dims the rest of the detail screen**: `_TrailerSection` now emits an `onPlayingChanged(bool)` callback when the inline YouTube player expands/collapses. The parent `_TitleDetailScreenState` owns `bool _trailerPlaying` and wraps every non-trailer ListView child in a `_dimmable()` helper — `AnimatedOpacity(0.3, 220ms)` + `IgnorePointer(ignoring: _trailerPlaying)` — so backdrop, poster/header, action row, overview, ratings section, AI blurb, household ratings, and similar-titles carousel all fade out + become un-tappable while the trailer plays. The trailer section itself is not wrapped so it stays at 100% opacity. Collapsing the trailer ("Hide trailer" button or popping the screen) restores the siblings. Don't forget to wrap any new sibling section you add — an unwrapped child will pop through the dim effect.

35. **External ratings (OMDb) + TMDB reviews**: `functions/src/externalRatings.ts` exposes a `fetchExternalRatings` callable that takes `{imdbId}` (validated against `/^tt\d{6,10}$/`), serves from `/externalRatings/{imdbId}` when fresh (7-day TTL), otherwise hits OMDb and caches. OMDb returns `{Response: "False"}` for unknown ids rather than a 404 — those are cached with `notFound: true` so we don't hammer OMDb for obscure titles. OMDb errors fall back to stale cache when available. Client: `ExternalRatingsService` (`lib/services/external_ratings_service.dart`) wraps the callable with a session-scoped in-memory memo so scrolling doesn't re-call for the same imdb id within one app run. `externalRatingsProvider` (family on imdbId) drives both the `_ScoreTile` row on title detail (IMDb / RT / Metacritic / TMDB) and the inline `_ImdbChip` on the `_RecCard` Home list rows. Tiles/chips are silent when the corresponding score is missing (title may be untracked by one or more services, or imdb_id not yet resolved for the rec doc). OMDb free tier caps at 1,000 req/day — the 7-day cache + per-imdb keying means real load is ≪1 per title per week across all users sharing the same cache. Firestore rules allow public read on `/externalRatings/{imdbId}` (public movie metadata, not household-scoped); writes are admin-SDK only. Reviews: TMDB's `/{mediaType}/{id}/reviews` endpoint is free (same TMDB key), exposed via `TmdbService.reviews`. `reviewsProvider` (family on `TitleRef(mediaType, tmdbId)`) sorts by content length desc so substantive reviews surface first. `_ReviewsSection` on title detail is collapsed by default (just a header + count); expanding shows up to 5 reviews as `_ReviewCard`s. Each card truncates at 320 chars with a "Read more" toggle. Silent on zero-review titles — don't reintroduce a "No reviews yet" placeholder; the reviews count in the header already communicates the state. Both sections are wrapped in `_dimmable()` so trailer-playback dims them.

36. **imdb_id stamping on rec docs (row-chip enabler)**: `Recommendation.imdbId` is lazily populated on `/households/{hh}/recommendations/{key}` so the Home list-row IMDb chip can render without an extra TMDB round-trip per row. Three write paths: (a) `RecommendationsService.writeCandidateDocs` returns the list of `(mediaType, tmdbId)` pairs that still need resolution — docs already carrying `imdb_id` are excluded. (b) `_backgroundResolveImdbIds` fires after Phase A completes, fanning out `TmdbService.externalIds(mediaType, tmdbId)` with concurrency=8 and writing `imdb_id` back via `.update()`. Errors are swallowed per-row. Independent of Claude scoring so a slow TMDB response can't delay rescoring. (c) `stampImdbId` — zero-cost opportunistic write invoked from `TitleDetail.initState` once the details future resolves (it already has the imdb_id via `_imdbIdFor`). Handles the "user opens a title before the background resolver finishes" case. `Recommendation.fromDoc` coerces empty-string `imdb_id` to `null` so a negative result doesn't keep retrying via a truthy check — but we still write the empty string to mark "resolved, no imdb entry" (TMDB returned a miss). `_RecCard` is a `ConsumerWidget` that watches `externalRatingsProvider(rec.imdbId!)` only when `imdbId != null` — no watch means no CF call, keeping the OMDb budget safe for the first-render wave before background resolution completes.
