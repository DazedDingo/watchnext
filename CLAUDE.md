# WatchNext — Claude.md

## Project

Shared movie & TV recommender for two-person households backed by Trakt integration and AI recommendations.

## Tech Stack

- **Frontend**: Flutter 3.11+ (Dart) with Material 3 dark theme
- **State management**: Riverpod 2.5 (providers + StateNotifier)
- **Routing**: go_router 14.2 with typed routes
- **Backend**: Firebase (Auth, Firestore, Cloud Functions, Cloud Messaging)
- **Cloud Functions**: Node 22 TypeScript, firebase-admin 13, firebase-functions 7, Anthropic SDK 0.30
- **External APIs**: 
  - Trakt (watch tracking, OAuth 2.0, history/ratings/recommendations)
  - TMDB (movie/TV metadata)
  - Claude API (Phase 7+, taste profiles)
  - Reddit JSON (Phase 7+, social proof)
- **Storage**: SharedPreferences (device-level mode toggle), Firestore (all household data)

## Key Conventions

### State Management
- **Providers**: `Provider<T>` for singletons, `StreamProvider<T>` for Firestore listeners, `FutureProvider<T>` for one-shot async
- **StateNotifier**: Used only for mutable user state (`ModeController`)
- **Naming**: `*Provider` for providers, `*Service` for singletons, `*Controller` for StateNotifier
- **Autodipose**: StreamProviders handling screen-scoped data use `.autoDispose` to clean up listeners on pop

### Firestore Collections & Rules
```
households/{householdId}/
  ├─ members/{uid}              # User ref + Trakt tokens + predict counters
                                # Counters: predict_total / predict_wins (legacy, pre-split context);
                                #           predict_total_solo / predict_wins_solo;
                                #           predict_total_together / predict_wins_together
                                # trakt_history_scope: 'shared'|'personal'|'mixed' (default mixed)
                                #   stamps Rating.context on imported Trakt ratings
  ├─ watchEntries/{mediaType:tmdbId}
  │  └─ episodes/{traktId}      # Per-ep timestamps
  ├─ ratings/{id}               # {uid, level, target_id, stars, tags, note, rated_at, pushed_to_trakt, context?}
                                # context: 'solo'|'together'|null (null = legacy / Trakt historical)
  ├─ watchlist/{scope:owner_or_shared:mediaType:tmdbId}
                                # {mediaType, tmdbId, scope, owner_uid?, addedAt, ...}
                                # scope: 'shared' (both members) | 'solo' (owner_uid only)
                                # shared → owner_or_shared = 'shared'; solo → owner_or_shared = uid
  ├─ predictions/{mediaType:tmdbId}
                                # {entries: {uid: {stars, skipped, submitted_at, context?}}, reveal_seen, ...}
                                # entry.context: 'solo'|'together'|null (determines counter routing)
  ├─ recommendations/{id}       # {mediaType, tmdbId, match_score, match_score_solo, ai_blurb, ...}
  ├─ tasteProfile/{id}          # {combined, per_user, per_user_solo, per_user_together, member_uids, updatedAt}
                                # per_user_solo[uid]: built from ratings with context='solo' OR null
                                # per_user_together[uid]: built from ratings with context='together' OR null
                                # per_user[uid]: cross-context (kept for back-compat + legacy consumers)
  ├─ decisionHistory/{id}       # "Decide" screen picks
  ├─ conciergeHistory/{id}      # Concierge chat logs
  ├─ collections/{id}           # Custom lists (Phase 10+)
  └─ issueBatches/{batchId}     # "Report an issue" queue docs. Per-submitter. `status`: pending|dispatched|cancelled. Client-writable only to set `status = cancelled`; the rest is admin-SDK-only.
users/{uid}/
  └─ {householdId}              # Pointer for O(1) household lookup post-auth
invites/{token}
  └─ Lookup table: {householdId, createdBy, expiresAt}
redditMentions/{id}             # Global; written by CF
rescoreQueue/{householdId}      # Background re-score marker. Admin-SDK only.
                                # {household_id, dirty_since, last_scored_at}
                                # dirty_since > last_scored_at → drain picks it up
households/{householdId}/badges/{badgeId}
                                # Server-persisted badge state. Admin-SDK writes only.
                                # {id, name, progress, target, earned, member_uid?, earned_at?, updated_at}
                                # badgeId = 'century_club' | 'genre_explorer' | 'prediction_machine_{uid}'
```
- All reads/writes scoped to household membership via `isMember(householdId)` check
- Two-person cap enforced client-side (`HouseholdService.joinByInviteCode`)
- Trakt OAuth tokens encrypted at `members/{uid}.trakt_refresh_token`

### Screen/Widget Structure
- **Screens**: Consume providers via `ref.watch()`, emit side effects via `ref.read().method()`
- **Shared widgets**: Stateless except `ScaffoldWithNavBar` (manages share intent listeners at root)
- **Error handling**: `AsyncErrorView` + per-provider error states + global error widget fallback
- **Bottom sheet**: `showModalBottomSheet()` for confirm dialogs; persistent on navigation
- **Share-to-Save** (Phase 4): `warm_start` + `cold_start` listeners in `ScaffoldWithNavBar` parse Android SEND intents

### External API Integration
- **Trakt**: Client ID as `--dart-define TRAKT_CLIENT_ID`, secret only in Firebase Secrets Manager
  - OAuth browser flow via `flutter_web_auth_2`, CSRF state saved to Firestore
  - Token refresh auto-triggered on 401; revoke on household leave
  - Callable proxies: `traktExchangeCode`, `traktRefreshToken`, `traktRevoke` in Cloud Functions
- **TMDB**: Key via `--dart-define TMDB_API_KEY`, fetches metadata on demand
- **Claude**: Async callable `concierge` + `generateTasteProfile`, AI blurbs cached in `/recommendations`

### Secret Handling
- Keys passed via `--dart-define-from-file=env.json` (build-time) or Firebase Secrets Manager (runtime)
- `env.json` gitignored; use `env.example.json` template
- Trakt `client_secret` lives **only** in `TRAKT_CLIENT_SECRET` Firebase secret
- No hardcoded secrets; all validated at startup

## Known Gotchas

1. **Cloud Functions region**: All CF callables explicitly set to `europe-west2` (London) to co-locate with Firestore; client code uses `FirebaseFunctions.instanceFor(region: 'europe-west2')` — do not default to us-central1
2. **Firestore model parsing**: Hardened to skip missing fields and map null → sensible defaults; see `fromDoc` factory methods. Partial documents from share-parser are normal
3. **Trakt token refresh**: Happens auto on 401; client must catch and retry. Token stored encrypted at member doc — check `firebase_auth_mocks` in tests for token mocking
4. **Two-person cap**: Enforced in service layer (line-of-code comment in rules acknowledges Firestore can't count siblings easily)
5. **Share-to-Save listeners**: Cold-start listener set up in `main.dart`, warm-start in `ScaffoldWithNavBar`; both parse Android SEND intents asynchronously
6. **Phase 7+ AI blurbs**: `Recommendation` doc has dual fields (`ai_blurb` for together, `ai_blurb_solo` for solo mode); only populated once taste engine runs
7. **TMDB genres**: Hardcoded map in `utils/tmdb_genres.dart` — update if TMDB API schema changes
8. **Debug keystore**: Pinned in `android/app/build.gradle`; SHA-1 verified in CI — restores on each checkout to keep Google Sign-In consistent
9. **ExpansionTile + flag-gated nested Consumer**: avoid `children: [if (_flag) Consumer(...)]`. The Consumer never observes the FutureProvider's resolution because the rebuild after `setState` races the tile's expansion animation; symptom is a spinner that never resolves. Instead split into a sticky-flag wrapper + separate `ConsumerWidget` body — see `_GenreRow`/`_GenreRowBody` in `discover_screen.dart`.
10. **Home filter stack**: seven knobs compose on the stream — genre set (free-form TMDB multi-select via `selectedGenresProvider`), runtime bucket, `YearRange {minYear?, maxYear?}`, media type (`MediaTypeFilter` via `mediaTypeFilterProvider`, per-mode persisted under `wn_media_type_solo/_together`), Awards (`awardsFilterProvider` — `AwardCategory?` of `bestPicture` | `palmeDor` | `baftaBestFilm`, per-mode persisted under `wn_awards_solo/_together`; see gotcha 35c for the baked-list architecture), Sort mode (`sortModeProvider` — one of `topRated` | `popularity` | `recent` | `underseen`, per-mode persisted under `wn_sort_mode_solo/_together`), and Curated source (`curatedSourceProvider` — `none` | `a24` (41077) | `neon` (90733) | `ghibli` (10342) | `searchlight` (43|127929), per-mode persisted under `wn_curated_source_solo/_together`; see gotcha 37b for id correctness + why Criterion was removed). Mood pills have been retired — the genre set is the canonical filter and mood presets were judged redundant with it. Exclude-animation has also been retired as a toggle — the baked Best Picture / Palme d'Or / BAFTA lists are curated free of anime so the redundant toggle was removed. In its place, `utils/animation_cap.dart`'s `capAnimation` runs as a client-side soft cap on Home (`kAnimationSoftCap=2`), applied AFTER every user filter in the Home pipeline but BEFORE dismissal/tonights-pick split. It's a no-op when `selectedGenres` contains `'Animation'` (explicit opt-in) and otherwise keeps at most 2 Animation-tagged rows in the visible pool. Rationale: `/trending/tv/week` + `/tv/top_rated` have no server-side without_genres support and both skew heavily anime on TMDB's global vote pool; without a cap, no-filter Home reads as "mostly anime" for households that aren't anime-first. This isn't applied to Decide, Upcoming, or Rewatch carousels — they have independent selection criteria. When the user narrows any way (genre/year/runtime/media type/awards/sort/curated), `recommendations_service.refresh()` fires `tmdbService.discoverPaged()` for the relevant media types; when media-type is set it only fires the matching one; when awards is set it splices the corresponding baked list (see gotcha 35c) into the merge order ahead of discover + trending so the `is_oscar_winner=true` tag sticks; when sort mode is `underseen` it passes `maxVoteCount=500` → `vote_count.lte=500` and suppresses the trending/top_rated baseline inside `buildCandidates`; `recent` picks a media-type-specific sort token (`primary_release_date.desc` for movie, `first_air_date.desc` for tv) via `SortMode.tmdbSortBy(mediaType)`; `popularity` uses `popularity.desc`; when curated source is set it passes `withCompanies` (e.g. `'41077'` for A24, `'43|127929'` for Searchlight's pre-Disney + current brand pipe-OR union) AND suppresses baseline (trending/top_rated would dilute a curator pool). Discover rows land in `buildCandidates` under `source='discover'` with `discoverCap=40` vs trending/top-rated's `tmdbCap=10`. The filter widgets live inside a single `_FiltersPanel` ExpansionTile on the Home screen. Panel body is flat (no horizontally-scrolling FilterChip rows): small uppercase section labels (`_FilterSectionLabel`), a `_MediaTypeBar` at the top (full-width 40px segmented toggle with movie/tv icons + "All/Movies/TV" labels + `selectionClick` haptic), a Genre InputChip row, the `YearRangeSlider`, a `_RuntimeSegment`, a `_SortModeSegment`, and two `_FilterDropdownShell`-backed popups (`_AwardsDropdown` + `_CuratedSourceSegment`) that theme their trigger when an active option is selected (primary 1.5px border, primary.withValues(alpha:0.08) background, bold primary label text). Include-watched remains a `_FilterSwitchRow`. Don't reintroduce per-option FilterChips or the old Oscar/Exclude-animation `Switch` rows — the dropdown-first layout was an intentional UX shift. Runtime filter is **strict**: `bucket.matches(r.runtime)` — unknown-runtime recs drop out. To keep the pool non-empty, `discoverPaged` uses TMDB's `with_runtime.gte/lte` server-side filter when a bucket is active, and the service stamps a representative runtime on each discover row (TMDB confirmed it's in-bounds). Trending/top_rated lack runtime and are intentionally excluded when a runtime bucket is active — we can't honestly show them under a specific length. Media type filter does strict equality `r.mediaType == filter.recMediaType`; cleared filter shows both. Year filter drops unknown-year items when any bound is set. Genre filter is **intersection**, not union: `selectedGenres.every(r.genres.contains)` — selecting Western + Sci-Fi shows only titles tagged with both, not "either." Rare AND intersections are deliberate (the user asked for something specific); unclassified recs (`r.genres=[]`) drop out under an active filter because "classed as all of these" can't be verified. Server-side `discoverPaged` still uses OR-joined `with_genres` + per-genre fallback (rung 2) so the raw pool + Claude scoring have plenty of signal; the client-side intersection does the final narrowing at display time. Same AND contract applies to `DecideFilters.matches` in Decide sessions. Awards filter is also strict: only recs whose `id` is in the selected award's baked-id set AND that carry `isOscarWinner=true` survive — trending/top_rated rows the current refresh didn't tag drop out even if they happen to be real winners. Sort mode + curated source don't add a client-side filter — they reshape the server-side pool; their contribution is carried entirely by the TMDB query params + baseline-suppression flags.
11. **SharedPreferences test caching**: `SharedPreferences.getInstance()` caches a singleton across tests in the same isolate, so calling `setMockInitialValues(...)` after an earlier test has already called `getInstance()` won't take effect on subsequent reads. Fix: after `setMockInitialValues`, either call `prefs.reload()` to refresh the cached instance, or construct a controller with an explicit prefs + override the StateNotifierProvider directly. See `test/widgets/genre_sheet_test.dart` for the override pattern (includes overriding `viewModeProvider` — otherwise `selectedGenresProvider` reads from the wrong mode).
12. **TMDB request timeout**: `TmdbService` wraps every `_get` in `.timeout(kRequestTimeout)` (15s default). `http.Client` has no built-in timeout and `discoverPaged` can fan out to ~18 TMDB calls on a single pull-to-refresh; any stuck request previously hung the refresh spinner indefinitely. Timeouts surface as `TimeoutException`, which the service layer's `_safeTmdb` wrapper swallows into an empty payload so a single flaky request doesn't blank the whole candidate pool. The constructor accepts a `timeout` override for tests.
13. **Two-phase refresh**: `RecommendationsService.refresh()` returns as soon as the Firestore batch write completes (Phase A — typically <5s); the `scoreRecommendations` callable + optional taste-profile refresh run in `unawaited(_backgroundScore(...))` (Phase B — 60-120s). Pull-to-refresh UX: spinner stops on Phase A so Home lights up immediately with default-scored recs; Claude's real scores arrive via Firestore stream updates later. New rec keys are seeded with `match_score=50, scored=false, ai_blurb=''`; existing keys skip those fields on merge so previously Claude-scored recs don't visibly regress. If Phase B fails (network, CF cold start, auth), the scheduled `processRescoreQueue` CF (every 10 min) picks up the slack — that's why `_backgroundScore` swallows errors into `developer.log`. **Widened pool**: `discoverPaged` defaults to `poolFloor=40, maxPages=5`; the CF `MAX_CANDIDATES=100` — upped from 20/3/50 so narrow filters (e.g. "War, 1970-1989") still yield a sizable list after client-side filters run. `drainPages` retries the next page on per-page errors (doesn't abort the whole rung).
14. **Stream window sized for Phase A**: `/recommendations` is streamed with `.limit(300)` — deliberately wider than the top-120 a user sees. Phase A writes fresh discover rows at `match_score=50`; if legacy Claude-scored (60–90) recs filled the first 120 slots, a narrow filter would see an empty list until Phase B re-ran. Wider window keeps just-written discover rows visible to the client-side filter from the instant Phase A lands. `writeCandidateDocs` reads up to 800 existing ids to preserve scores on merge.
15. **Home auto-refresh on filter change**: the Home screen's `build()` registers `ref.listen`s on `selectedGenresProvider`, `runtimeFilterProvider`, `yearRangeProvider`, `mediaTypeFilterProvider`, `awardsFilterProvider`, `sortModeProvider`, and `curatedSourceProvider`. Any change schedules a 700ms-debounced non-forcing refresh (`refreshRecommendationsProvider(false)`) so the pool is rebuilt against the current filter state without the user having to pull-to-refresh. The debounce Timer is cancelled in `dispose()`. Compare sets with `setEquals` from `package:flutter/foundation.dart` — the raw `!=` comparison on `Set` is reference equality and would fire spuriously. **Explicit-commit CTA** (`_ApplyFiltersCta` at the bottom of `_FiltersPanel`): Liquid-UI-flavoured primary button ("Show recommendations" + sparkle icon) that cancels the pending debounce, fires a non-forcing refresh immediately, and collapses the panel in one action. Rationale: when users toggle 3-4 filters in a row they want to commit the whole batch + see results now, not wait 700ms after the last toggle and then scroll past the still-expanded panel. The button uses the same accent gradient + two-layer glow as `LiquidSegmentedButton`'s selection blob so it reads as "the primary action in this panel" without looking like a stock Material button. `selectionClick` haptic fires on tap.

15b. **Deliberate pull-to-refresh**: `RefreshIndicator` on Home uses `displacement: 140` (default is 40) to require a substantial pull — accidental scroll overshoots no longer trigger a refresh, which burns TMDB + Claude budget. On trigger, a SnackBar shows a spinner + "Refreshing recommendations…" as clear visual confirmation; the `onRefresh` callback awaits `Future.wait([refresh, Future.delayed(1200ms)])` so the snackbar stays visible for a minimum of 1.2s even when Phase A completes faster. `clearSnackBars()` is called at both start and success so repeated refreshes don't stack multiple banners. `strokeWidth: 3.5` thickens the spinner for added visual weight. Don't lower `displacement` — the gesture is intentionally heavy to match the cost of the refresh (Phase A fans out to ~18 TMDB calls + a background Claude scoring call).
16. **Concierge SDK path**: `@anthropic-ai/sdk@0.30.1` doesn't accept `cache_control` on the stable `messages.create` `TextBlockParam` — only the deprecated `beta.promptCaching.messages.create` path does. Concierge is low-frequency (one chat at a time) so we intentionally use the stable endpoint without `cache_control` rather than ride the beta namespace. The Claude call is wrapped in try/catch so upstream errors surface as an `HttpsError("internal", <reason>)` with the original error message; the client parses `FirebaseFunctionsException` and shows the reason in the chat bubble instead of a generic "something went wrong".
18. **Callable result maps on Android**: `HttpsCallableResult.data` comes back as `Map<Object?, Object?>` from the Android platform channel (iOS varies too). `result.data as Map<String, dynamic>` compiles but throws `type 'Map<Object?, Object?>' is not a subtype of type 'Map<String, dynamic>'` at runtime; `.cast<Map<String, dynamic>>()` on a list of nested maps throws the same at iteration time. Always normalize with `Map<String, dynamic>.from(result.data as Map)` at the top level AND `Map<String, dynamic>.from(m)` per nested map before constructing models. `ConciergeService.chat` hit this only on Like-these responses (chat-only paths often have empty titles arrays, hiding the bug).
19. **"More like these"**: `/home/ubuntu/projects/watchnext/lib/screens/like_these/like_these_sheet.dart` reuses the `concierge` CF with a synthetic prompt — no separate endpoint. The sheet collects 2–8 TMDB seed titles via `searchMulti`, builds a "suggest titles that fit this group as a whole" message, and sends it with a fresh `sessionId` (so the concierge doesn't treat it as a continuation of an earlier chat) and empty `history`. Response titles render as tappable tiles that navigate to `/title/:mediaType/:tmdbId`. Entry point: "Like these" OutlinedButton in the Home action row next to "Surprise me". Also see TitleDetail's **Similar titles** carousel, which surfaces TMDB's `/similar` results (free — already appended via `append_to_response=similar`).

24. **TMDB merge order — discover leads, baseline follows**: `buildCandidates` puts `discoverMoviesPayload` and `discoverTvPayload` AHEAD of trending/top_rated in the dedup walk. Reason: when a row appears in both discover and trending, the discover tag carries the user's narrowing intent (genre, year, runtime, oscar keyword) — and for the Oscar filter specifically, the discover row carries `is_oscar_winner=true`. If trending won dedup, a current-year Best Picture that's also trending this week would lose the oscar tag and disappear under the Oscar filter. Tests in `recommendations_service_test.dart` (`buildCandidates — discover payloads`, `buildCandidates — oscar discover tagging`) lock this contract.

25. **Sticky `is_oscar_winner` + `curator` on merge**: `writeCandidateDocs` only writes the `is_oscar_winner` field when the candidate map carries `true`. False/missing values are intentionally omitted so Firestore's `merge: true` semantics leave any previously-written `true` untouched — a movie doesn't un-win an Oscar between refreshes. Mirror logic in `buildCandidates`: it only stamps `is_oscar_winner: true` on discover rows when `discoverIsOscar=true`, never `false`. Same pattern for `curator` (e.g. `'criterion'`): set only when `discoverCurator` is non-empty on a discover row; empty-string and missing are both ignored by `writeCandidateDocs` so a prior `curator: 'criterion'` survives merge with a non-curator refresh. Both tags drive strict client-side filters on Home — Oscar filter keeps only `isOscarWinner=true`, Curated filter keeps only `curator == CuratedSource.name` — which prevents stale non-matching recs from leaking through when a filter is active. So the round trip from "user toggled curator on → refresh → tag written" is preserved across subsequent refreshes that don't include the keyword/company filter.

27. **Firestore dot-notation is `.update()`-only**: `ref.set({'a.b': v}, SetOptions(merge: true))` stores a **literal field named "a.b"** with a dot in it — it does NOT write the nested path `a.b`. Only `ref.update({'a.b': v})` interprets the dot as a nested path. This bit `WatchEntryService` twice (mark watched didn't persist; unmark watched didn't clear) before it was caught. If you need to mutate a nested map field, use `.update()`. If the doc may not exist, `.get()` first and early-return — `update()` throws on missing docs, whereas `set(merge:true)` silently creates them but won't honour dot-notation. Tests in `test/services/watch_entry_service_test.dart` explicitly assert `doc.containsKey('watched_by.<uid>') == false` so a regression to `.set(merge:true)` fails the suite loudly.

31. **Refresh-race epoch guard**: `RecommendationsService` owns an `int _refreshEpoch = 0` counter. At the top of `refresh()` we capture `final myEpoch = ++_refreshEpoch;`. Before the Firestore write AND before kicking off background Claude scoring we check `if (myEpoch != _refreshEpoch) return;` — a newer refresh has superseded this one, so bail silently. Without this guard, an older refresh whose TMDB fan-out (18 calls on a narrow filter) returned slowly could complete its `writeCandidateDocs` after a newer refresh's already-committed pool, overwriting the fresher results — the "selected filters show a set of options that get overwritten a few seconds later" symptom. Both refreshes still run their fetches to completion; we only short-circuit the mutation side so the *last-started* refresh wins regardless of which finishes TMDB first. Tests in `test/services/recommendations_service_test.dart` under `RecommendationsService.refresh — epoch guard` inject a slow-returning TMDB mock for refresh 1 and a fast one for refresh 2, then assert refresh 1's candidates never land in Firestore.

35b. **OMDb auth/quota errors are NOT cached**: `fetchExternalRatings` distinguishes OMDb's "Invalid API key" / "Daily limit" / "Request limit" responses from genuine "movie not found" ones. Auth/quota → `throw HttpsError("internal", ...)` so the CF fails and the client renders no chip. Not-found → `{notFound: true}` cached for 7 days. Why it matters: the original implementation cached everything-with-`Response=False` as notFound, which meant a single spell of bad auth (or a key that hadn't been activated yet) would poison the cache for a week even after the key started working. If you're debugging "chips not rendering", `curl https://www.omdbapi.com/?i=tt0068646&apikey=$KEY` first — `{Response: "False", Error: "Invalid API key!"}` means the key isn't activated (OMDb emails an activation link after signup; users sometimes miss it). Purge stale cache with `firebase firestore:delete externalRatings --recursive --force`. Also: `fetchFromOmdb` defensively `.trim()`s the secret before building the URL — Secret Manager values pasted through a terminal commonly pick up a trailing newline, which `URLSearchParams.set` encodes as `%0A` and OMDb rejects as "Invalid API key!" even when the underlying key is valid. Verify with `firebase functions:secrets:access OMDB_API_KEY 2>&1 | od -c` — trailing `\n` means the stored value is polluted; re-set via `printf '%s' 'KEY' | firebase functions:secrets:set OMDB_API_KEY --data-file=-` and redeploy to pin the new version.

36. **Narrow-filter auto-widen**: `isNarrowFilterCombo` in `recommendations_service.dart` (pure helper, tested) counts orthogonal narrowing filters — genre set, year range, runtime bucket, awards (any category), curated source, underseen sort. Two or more → the refresh's `discoverPaged` call swaps defaults (`poolFloor=40, maxPages=5, minVoteCount=50, discoverCap=40`) for wider (`100 / 10 / 10 / 100`). Broad queries stay on the narrow budget. Why not just always widen: broad queries hit `poolFloor` on page 1, so the extra budget is pure waste; only narrow combos benefit. `discoverCap` must move in lockstep with `poolFloor` — otherwise `buildCandidates` would cap the pool at 40 and throw away the extra fetched rows. `MediaTypeFilter` doesn't count toward narrowness — it gates *which* side of the discover fan-out fires, not how deep it goes. `SortMode.popularity` / `.recent` reshape ranking without thinning the pool, so they're also excluded; only `.underseen` (which caps `vote_count.lte`) contributes. Four write paths: (a) `RecommendationsService.writeCandidateDocs` returns the list of `(mediaType, tmdbId)` pairs that still need resolution — docs already carrying `imdb_id` are excluded. (b) `_backgroundResolveImdbIds` fires after Phase A completes, fanning out `TmdbService.externalIds(mediaType, tmdbId)` with concurrency=8 and writing `imdb_id` back via `.update()`. Errors are swallowed per-row. Independent of Claude scoring so a slow TMDB response can't delay rescoring. (c) `stampImdbId` — zero-cost opportunistic write invoked from `TitleDetail.initState` once the details future resolves (it already has the imdb_id via `_imdbIdFor`). Handles the "user opens a title before the background resolver finishes" case. (d) `backfillMissingImdbIds(householdId, limit=300)` — public one-shot that reads the top-N rec docs and fires `_backgroundResolveImdbIds` on any missing `imdb_id`. Called from `_HomeScreenState.build` behind a `_imdbBackfillStarted` session guard so Home-open is enough to start resolution on fresh installs (without this, the feature looked broken until the user pulled-to-refresh). Cost at steady state: one Firestore `.limit(300).get()` + zero TMDB calls once everything's stamped. `Recommendation.fromDoc` coerces empty-string `imdb_id` to `null` so a negative result doesn't keep retrying via a truthy check — but we still write the empty string to mark "resolved, no imdb entry" (TMDB returned a miss). `_RecCard` is a `ConsumerWidget` that watches `externalRatingsProvider(rec.imdbId!)` only when `imdbId != null` — no watch means no CF call, keeping the OMDb budget safe for the first-render wave before background resolution completes.

42. **Liquid UI family**: three pieces share one visual language so the app chrome reads as a set: (a) `LiquidNavBar` (see `app.dart` entry above). (b) `LiquidSegmentedButton<T>` (`lib/widgets/liquid_segmented_button.dart`) — equal-width Row of segments with an `AnimatedPositioned` gradient blob that slides under the selected label (320ms, `easeOutCubic`), same accent gradient + softer single-layer shadow vs the nav bar's two-layer glow (dialled down because segmented controls live inside panels, not at the screen edge). Used by `ModeToggle` (AppBar Solo/Together) and the home filter panel's `_MediaTypeBar` / `_RuntimeSegment` / `_SortModeSegment` — M3's stock `SegmentedButton` was retired across those sites. Two density presets: `compact` (34px, 12pt) for in-panel filters, `standard` (40px, 13pt) for prominent bars. Don't bring back raw `SegmentedButton` for new inline selectors — use `LiquidSegmentedButton` so the family stays coherent. (c) `AppBarTheme.shape` — `themeDataProvider` paints a 0.5px `outlineVariant` hairline at the bottom of every AppBar (matching the nav bar's top-edge hairline). The pair frames the content area top+bottom. No gradient on the AppBar itself — just the divider — because AppBars with a `bottom: TabBar` (Library) + custom titles (animated WatchNext wordmark on Home/Login) would fight a gradient background but tolerate the hairline cleanly. If a future AppBar needs to suppress the divider (rare), override `shape:` per-instance.

43. **Keyword-driven genre augmentation**: TMDB's canonical genre taxonomy is narrow — Starship Troopers is `{Action, Adventure, Science Fiction}`, not `{Science Fiction, War}` — so AND-intersection filters like "Sci-Fi + War" zero out on cross-genre titles. `utils/keyword_genre_augment.dart`'s `augmentGenresWithKeywords` fills the gap: a `const Map<int, Set<String>> kKeywordToExtraGenres` maps TMDB keyword ids → extra genre names, and the augmenter unions those in without ever removing canonical genres (additive only — a Comedy stays a Comedy). **Three write paths** (mirror of gotcha 41's imdb_id stamping): (a) `RecommendationsService.stampAugmentedGenres` — opportunistic write from `TitleDetail.initState` once the details future resolves. Zero extra TMDB calls because the details payload already appends `keywords`. (b) `_backgroundAugmentGenres` — fires post-Phase-A inside `refresh()` via `unawaited(backfillMissingAugmentedGenres(householdId))`, concurrency=8, fetches `/{mediaType}/{id}/keywords` for every rec doc missing the `keywords_fetched` flag. (c) `backfillMissingAugmentedGenres(householdId, limit=300)` — public one-shot fired from `_HomeScreenState.build` behind a `_keywordsBackfillStarted` session guard so Home-open enriches cold pools on fresh installs without pull-to-refresh. Movies carry `keywords: [...]`; TV carries `results: [...]` — all callers coalesce both shapes. Writes are `.update({'keywords_fetched': true, 'genres': augmented?})` with the flag written unconditionally on success (so we don't retry forever on rows with no mapped keywords) and the genres field written only when the set actually grew. Missing rec docs + TMDB errors are silently swallowed — augmentation is a nice-to-have, not UI-critical. `Recommendation.fromDoc` is unchanged — it just reads `genres` as a list, and an augmented list stays valid. **Caveat: sticky flag**: `keywords_fetched: true` prevents re-hitting TMDB for the same rows, so adding new entries to `kKeywordToExtraGenres` won't re-augment existing docs. Mitigation options (not shipped): (1) bump a `keywords_version` field and re-fetch when the stored version is older; (2) manually clear the flag via an admin tool. **Populating the map**: `scripts/generate_keyword_genre_map.py` takes a hand-curated `(name, [genres])` seed list, queries `/search/keyword`, emits paste-ready Dart entries. Re-run whenever new seeds are added. Current seed cross-genre focus: sci-fi/war (space marines, alien invasion, space battle), westerns (space western, neo-western), horror subgenres (slasher, cosmic horror, zombie, found footage), crime crossovers (neo-noir, heist), sci-fi flavours (cyberpunk, time travel, kaiju). Map is conservative — only keywords that reliably imply the extra genre; broad tags like "fight" or "based on novel" are rejected as noise. Existing starter map entries (9951 alien, 4458 post-apocalyptic, 4565 dystopia, 10084 anime, 9715 superhero) are seeds until the generator runs against a live TMDB key.
