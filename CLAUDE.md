## Project

Shared movie & TV recommender for two-person households backed by Trakt integration and AI recommendations.

## Tech Stack

- **Frontend**: Flutter 3.11+ (Dart) with Material 3 dark theme
- **State management**: Riverpod 2.5 (providers + StateNotifier)
- **Routing**: go_router 14.2 with typed routes
- **Backend**: Firebase (Auth, Firestore, Cloud Functions, Cloud Messaging)
- **Cloud Functions**: Node 22 TypeScript, firebase-admin 13, firebase-functions 7, Google Generative AI SDK 0.24 (gemini-2.5-flash)
- **External APIs**: 
  - Trakt (watch tracking, OAuth 2.0, history/ratings/recommendations)
  - TMDB (movie/TV metadata)
  - Gemini API (Phase 7+, taste profiles + concierge â€” free tier, 1,500 req/day)
  - Reddit JSON (Phase 7+, social proof)
- **Storage**: SharedPreferences (device-level mode toggle), Firestore (all household data)

## Operating Cost

Designed to run at $0/mo for a two-person household; every dependency sits on a free tier sized well above household-scale traffic. If a future change would push past one of these limits, that's the moment to call cost out â€” until then, monthly run cost is effectively zero.

- **Firebase (Blaze plan, required for outbound HTTPS from Cloud Functions)** â€” Spark-equivalent free quotas still apply: Firestore 50K reads / 20K writes / 1GB storage per day, Cloud Functions 2M invocations + 400K GB-seconds per month, Auth + FCM unmetered. Real usage for two members across all current features sits well under each â€” typical weekly bill is $0.
- **TMDB** â€” free, key-only auth; no rate limit issue at household scale. Used for everything: details, search, discover (with the wide narrow-filter budget â€” gotcha 36), trending, upcoming, season/episode payloads.
- **Trakt** â€” free OAuth client; rate-limited but the per-user sync polling stays well below the threshold.
- **Reddit JSON** â€” public no-auth feed; scraped server-side only.
- **Gemini 2.5 Flash** â€” AI Studio free tier, 1,500 req/day. Drives `scoreRecommendations`, `rescoreRecommendations`, and `concierge`. A two-person household generates well under the daily cap; see gotcha 16 + 15c (state-hash dedupe is the safety belt that prevents a stuck refresh from burning the budget).
- **OMDb** â€” free tier 1,000 req/day; a 7-day Firestore cache + per-imdb keying means real load is â‰ª 1 per title per week. See gotcha 35.
- **GitHub Releases** â€” free distribution channel; no Play Store fee. See `feedback_watchnext_distribution.md` memory.
- **Domain/hosting** â€” none; the app is a sideload APK + Firebase backend. No web frontend.

**When a change adds cost**, flag it in the PR description / proposal *before* implementation: which service, which quota, and the projected delta. Examples that would warrant a flag: a new always-on Cloud Functions schedule firing more than once a minute, a model swap to Gemini Pro, an OMDb path that bypasses cache, a Firestore listener fanout that scales with collection size. Examples that don't: a new Riverpod provider, a new screen, a new TMDB method (TMDB is free), a new Firestore field on an existing doc.

## Known Gotchas

1. **Cloud Functions region**: All CF callables explicitly set to `europe-west2` (London) to co-locate with Firestore; client code uses `FirebaseFunctions.instanceFor(region: 'europe-west2')` â€” do not default to us-central1
2. **Firestore model parsing**: Hardened to skip missing fields and map null â†’ sensible defaults; see `fromDoc` factory methods. Partial documents from share-parser are normal
3. **Trakt token refresh**: Happens auto on 401; client must catch and retry. Token stored encrypted at member doc â€” check `firebase_auth_mocks` in tests for token mocking
4. **Two-person cap**: Enforced in service layer (line-of-code comment in rules acknowledges Firestore can't count siblings easily)
5. **Share-to-Save listeners**: Cold-start listener set up in `main.dart`, warm-start in `ScaffoldWithNavBar`; both parse Android SEND intents asynchronously
6. **Phase 7+ AI blurbs**: `Recommendation` doc has dual fields (`ai_blurb` for together, `ai_blurb_solo` for solo mode); only populated once taste engine runs
7. **TMDB genres**: Hardcoded map in `utils/tmdb_genres.dart` â€” update if TMDB API schema changes
8. **Debug keystore**: Pinned in `android/app/build.gradle`; SHA-1 verified in CI â€” restores on each checkout to keep Google Sign-In consistent
9. **ExpansionTile + flag-gated nested Consumer**: avoid `children: [if (_flag) Consumer(...)]`. The Consumer never observes the FutureProvider's resolution because the rebuild after `setState` races the tile's expansion animation; symptom is a spinner that never resolves. Instead split into a sticky-flag wrapper + separate `ConsumerWidget` body â€” see `_GenreRow`/`_GenreRowBody` in `discover_screen.dart`.
10. **Home filter stack**: seven knobs compose on the stream â€” genre set (free-form TMDB multi-select via `selectedGenresProvider`), runtime bucket, `YearRange {minYear?, maxYear?}`, media type (`MediaTypeFilter` via `mediaTypeFilterProvider`, per-mode persisted under `wn_media_type_solo/_together`), Awards (`awardsFilterProvider` â€” non-nullable `AwardCategory` of `none` (default) | `any` | `bestPicture` | `palmeDor` | `baftaBestFilm` | `goldenGlobeDrama` | `goldenGlobeMusicalComedy`, per-mode persisted under `wn_awards_solo/_together`; see gotcha 35c for the baked-list architecture + the PopupMenuButton-null-value bug that drove the non-nullable refactor), Sort mode (`sortModeProvider` â€” one of `topRated` | `popularity` | `recent` | `underseen`, per-mode persisted under `wn_sort_mode_solo/_together`), and Curated source (`curatedSourceProvider` â€” `none` | `a24` (41077) | `neon` (90733) | `ghibli` (10342) | `searchlight` (43|127929), per-mode persisted under `wn_curated_source_solo/_together`; see gotcha 37b for id correctness + why Criterion was removed). Mood pills have been retired â€” the genre set is the canonical filter and mood presets were judged redundant with it. Exclude-animation has also been retired as a toggle â€” the baked Best Picture / Palme d'Or / BAFTA lists are curated free of anime so the redundant toggle was removed. In its place, `utils/animation_cap.dart`'s `capAnimation` runs as a client-side soft cap on Home (`kAnimationSoftCap=2`), applied AFTER every user filter in the Home pipeline but BEFORE dismissal/tonights-pick split. It's a no-op when `selectedGenres` contains `'Animation'` (explicit opt-in) and otherwise keeps at most 2 Animation-tagged rows in the visible pool. Rationale: `/trending/tv/week` + `/tv/top_rated` have no server-side without_genres support and both skew heavily anime on TMDB's global vote pool; without a cap, no-filter Home reads as "mostly anime" for households that aren't anime-first. This isn't applied to Decide, Upcoming, or Rewatch carousels â€” they have independent selection criteria. When the user narrows any way (genre/year/runtime/media type/awards/sort/curated), `recommendations_service.refresh()` fires `tmdbService.discoverPaged()` for the relevant media types; when media-type is set it only fires the matching one; when awards is set it splices the corresponding baked list (see gotcha 35c) into the merge order ahead of discover + trending so the `is_oscar_winner=true` tag sticks; when sort mode is `underseen` it passes `maxVoteCount=500` â†’ `vote_count.lte=500` and suppresses the trending/top_rated baseline inside `buildCandidates`; `recent` picks a media-type-specific sort token (`primary_release_date.desc` for movie, `first_air_date.desc` for tv) via `SortMode.tmdbSortBy(mediaType)`; `popularity` uses `popularity.desc`; when curated source is set it passes `withCompanies` (e.g. `'41077'` for A24, `'43|127929'` for Searchlight's pre-Disney + current brand pipe-OR union) AND suppresses baseline (trending/top_rated would dilute a curator pool). Discover rows land in `buildCandidates` under `source='discover'` with `discoverCap=40` vs trending/top-rated's `tmdbCap=10`. The filter widgets live inside a single `_FiltersPanel` ExpansionTile on the Home screen. Panel body is flat (no horizontally-scrolling FilterChip rows): small uppercase section labels (`_FilterSectionLabel`), a `_MediaTypeBar` at the top (full-width 40px segmented toggle with movie/tv icons + "All/Movies/TV" labels + `selectionClick` haptic), a Genre InputChip row, the `YearRangeSlider`, a `_RuntimeSegment`, a `_SortModeSegment`, and two `_FilterDropdownShell`-backed popups (`_AwardsDropdown` + `_CuratedSourceSegment`) that theme their trigger when an active option is selected (primary 1.5px border, primary.withValues(alpha:0.08) background, bold primary label text). Include-watched remains a `_FilterSwitchRow`. Don't reintroduce per-option FilterChips or the old Oscar/Exclude-animation `Switch` rows â€” the dropdown-first layout was an intentional UX shift. Runtime filter is **strict**: `bucket.matches(r.runtime)` â€” unknown-runtime recs drop out. To keep the pool non-empty, `discoverPaged` uses TMDB's `with_runtime.gte/lte` server-side filter when a bucket is active, and the service stamps a representative runtime on each discover row (TMDB confirmed it's in-bounds). Trending/top_rated lack runtime and are intentionally excluded when a runtime bucket is active â€” we can't honestly show them under a specific length. Media type filter does strict equality `r.mediaType == filter.recMediaType`; cleared filter shows both. Year filter drops unknown-year items when any bound is set. Genre filter is **intersection**, not union: each selected name must be satisfied by every surviving rec. Implemented via `genreMatches(r.genres, g)` (from `utils/tmdb_genres.dart`) â€” a direct `contains` check plus a fall-back through `kGenreSynonyms`. TMDB splits genres into separate movie + TV vocabularies that don't line up concept-for-concept: movies have `Science Fiction` (878) / `War` (10752); TV lumps these as `Sci-Fi & Fantasy` (10765) / `War & Politics` (10768); TV has `Kids` (10762) for what movies call `Family` (10751); TV has `Action & Adventure` (10759) for movies' separate `Action` (28) + `Adventure` (12). `allGenresProvider` unions both label sets into one picker, so picks used to cross the taxonomy divide and zero out the AND intersection ("Sci-Fi + Kids returns nothing" class of bug). `kGenreSynonyms` declares concept-level equivalence â€” applied at BOTH `genreIdsFromNames` (so discover pools include the equivalent ids on each media type) AND `genreMatches` (so a rec tagged `["Sci-Fi & Fantasy", "War & Politics"]` satisfies a `["Science Fiction", "War"]` pick). Applying at both layers is load-bearing â€” the fetch side populates the pool, the filter side lets cross-taxonomy recs through. Rare AND intersections are deliberate (the user asked for something specific); unclassified recs (`r.genres=[]`) drop out under an active filter because "classed as all of these" can't be verified. Server-side `discoverPaged` uses a six-rung fallback ladder to match the client-side AND intersection: **(1)** AND-join (`,`) + year â†’ strict intersection; **(2)** OR-join (`|`) + year â†’ union that feeds the keyword augmenter; **(3)** per-genre + year â†’ rescue sparse single genres; **(4)** AND no-year; **(5)** OR no-year; **(6)** per-genre no-year. Single-genre queries skip rungs 1/3/4/6 (AND â‰¡ OR for one genre). The AND-first ordering fixes cross-genre pool density ("Sci-Fi + War" without AND-first returned Schindler's List + Interstellar instead of Starship Troopers + Overlord). Same AND contract + synonym expansion applies to `DecideFilters.matches` in Decide sessions. Awards filter is also strict: only recs whose `id` is in the selected award's baked-id set AND that carry `isOscarWinner=true` survive â€” trending/top_rated rows the current refresh didn't tag drop out even if they happen to be real winners. Sort mode + curated source don't add a client-side filter â€” they reshape the server-side pool; their contribution is carried entirely by the TMDB query params + baseline-suppression flags.
11. **SharedPreferences test caching**: `SharedPreferences.getInstance()` caches a singleton across tests in the same isolate, so calling `setMockInitialValues(...)` after an earlier test has already called `getInstance()` won't take effect on subsequent reads. Fix: after `setMockInitialValues`, either call `prefs.reload()` to refresh the cached instance, or construct a controller with an explicit prefs + override the StateNotifierProvider directly. See `test/widgets/genre_sheet_test.dart` for the override pattern (includes overriding `viewModeProvider` â€” otherwise `selectedGenresProvider` reads from the wrong mode).
12. **TMDB request timeout**: `TmdbService` wraps every `_get` in `.timeout(kRequestTimeout)` (15s default). `http.Client` has no built-in timeout and `discoverPaged` can fan out to ~18 TMDB calls on a single pull-to-refresh; any stuck request previously hung the refresh spinner indefinitely. Timeouts surface as `TimeoutException`, which the service layer's `_safeTmdb` wrapper swallows into an empty payload so a single flaky request doesn't blank the whole candidate pool. The constructor accepts a `timeout` override for tests.
13. **Two-phase refresh**: `RecommendationsService.refresh()` returns as soon as the Firestore batch write completes (Phase A â€” typically <5s); the `scoreRecommendations` callable + optional taste-profile refresh run in `unawaited(_backgroundScore(...))` (Phase B â€” 60-120s). Pull-to-refresh UX: spinner stops on Phase A so Home lights up immediately with default-scored recs; Claude's real scores arrive via Firestore stream updates later. New rec keys are seeded with `match_score=50, scored=false, ai_blurb=''`; existing keys skip those fields on merge so previously Claude-scored recs don't visibly regress. If Phase B fails (network, CF cold start, auth), the scheduled `processRescoreQueue` CF (every 10 min) picks up the slack â€” that's why `_backgroundScore` swallows errors into `developer.log`. **Widened pool**: `discoverPaged` defaults to `poolFloor=40, maxPages=5`; the CF `MAX_CANDIDATES=100` â€” upped from 20/3/50 so narrow filters (e.g. "War, 1970-1989") still yield a sizable list after client-side filters run. `drainPages` retries the next page on per-page errors (doesn't abort the whole rung).
14. **Stream window sized for Phase A**: `/recommendations` is streamed with `.limit(600)` â€” deliberately wider than the top-120 a user sees. The rec collection accumulates over months of refreshes, so the original 300 wasn't enough once cumulative Claude-scored docs crowded the top â€” 600 gives narrow-filter combos (e.g. "Sci-Fi + Western on TV") room to land in-window from the moment Phase A commits. 14b. **Rec collection prune (kRecCollectionCap=800)**: `writeCandidateDocs` ends with a `_pruneRecsBeyondCap` pass that caps the `/recommendations` collection at 800 docs, sized as `streamLimit (600) + headroom (~200)` so freshly-written narrow-filter rows reliably land inside the visible stream window even after months of accumulated refreshes. Without this, the home stream's `.limit(600).orderBy(match_score desc)` silently drops new 50-scored rows below the cutoff once cumulative Claude-scored (60â€“90) docs exceed 600. Walk: `col.orderBy('generated_at').get()` (oldest-first), iterate up to `(size - cap)` rows, delete only those that pass NONE of the protections â€” watchlist (`source='watchlist'`), sticky-tagged (`is_oscar_winner=true` OR non-empty `curator`, mirroring gotcha 25), just-written keys (Claude is about to score them), high-confidence (`match_score >= 80`, OR `scored=true && >= 70`). Awaited inside `writeCandidateDocs` (after the merge batches commit, before returning the imdb-needs list) so concurrent writes can't race the delete walk. Errors are swallowed â€” pruning is hygiene, not part of the user-facing refresh contract. Chunked into 450-op `WriteBatch` writes (Firestore's 500-op cap). The cumulative protection set means a household with 800 high-scored + saved + award-tagged recs can grow past the cap (no eligible rows to delete); that's an acceptable trade-off vs. accidentally trimming user-curated content. Tests in `writeCandidateDocs â€” prune` lock each protection in isolation + the 1500-doc-seed â†’ 800 trim guarantee.
